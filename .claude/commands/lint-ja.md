# 自然な日本語ドキュメントを書く

ドキュメントを作成・編集する際は、以下のガイドラインに従ってください。

## 避けるべきパターン

### 1. リストの機械的なフォーマット

**NG例:**

```markdown
- **重要**: これは重要な項目です
- **注意**: 注意が必要な項目です
- ✅ 完了した項目
- ❌ 失敗した項目
- 💡 アイデア項目
```

**OK例:**

```markdown
- 重要な項目: これは重要な項目です
- 注意事項: 注意が必要な項目です
- 完了した項目
- 失敗した項目
- アイデア項目
```

### 2. 誇張表現・ハイプ的な表現

**NG例:**

- 革命的な技術
- ゲームチェンジャー
- 世界初の
- 究極の
- 完全に〜を解決
- すべての課題を解決
- 魔法のように
- 奇跡的な
- パラダイムシフト
- 次世代の

**OK例:**

- 効果的な技術
- 大きな変化をもたらす
- 新しい
- 高い〜
- 多くの問題を解決
- 主要な課題を解決
- スムーズに
- 優れた
- 大きな変化
- 新しい

### 2-2. 攻撃的・煽情的な表現

競合比較や戦略記述で、攻撃的・感情的な表現を避ける。事実ベースで淡々と書く。

**NG例:**

- 圧倒する
- 地獄
- 武器になる
- 撒き餌
- 叩き潰す
- 殴り込む
- 刺さる

**OK例:**

- 優位性がある / 差別化できる
- 負担が大きい / 工数がかかる
- 強みになる / 有効に機能する
- 導入のきっかけ / フック
- シェアを取る
- 参入する
- 響く / 訴求力がある

### 3. 機械的な強調パターン

**NG例:**

```markdown
これは**非常に**重要な項目です。
**注意**してください。

# **重要なお知らせ**

## **太字**の見出し
```

**OK例:**

```markdown
これは重要な項目です。
注意してください。

# 重要なお知らせ

## 太字の見出し
```

### 3-2. 太字の使いどころ

太字は、用語を初めて定義・導入するときと、論理の要所（誤読を防ぐ否定や節の帰結）に限って使う。1つの節で1〜2箇所までを目安にする。すでに導入した用語を話題として指すときや、引用・通称には、太字ではなく「」を使う。

### 4. 絵文字の使用禁止

ドキュメント内で絵文字を使用しない。

**NG例:**

```markdown
✅ 完了
❌ 失敗
💡 ヒント
🚀 リリース
⚠️ 警告
📝 メモ
🔥 重要
```

**OK例:**

```markdown
- 完了
- 失敗
- ヒント
- リリース
- 警告
- メモ
- 重要
```

### 5. ダッシュ（--）による挿入・補足

英語的なダッシュ（`--` や `—`）を文中の挿入・補足に使わない。日本語では句点で文を区切るか、括弧で補足する。

**NG例:**

```markdown
6 つ目のバージョンが必要になった -- そうした場面で回収できます。
この設計は柔軟で — 将来の拡張にも対応できます。
```

**OK例:**

```markdown
6 つ目のバージョンが必要になった。そうした場面で回収できます。
この設計は柔軟で、将来の拡張にも対応できます。
```

### 5-2. 見出しの「トピック：補足」「トピック -- 補足」パターン

見出しを「概念：説明」「概念 -- 説明」のように機械的に分割しない。自然な日本語のフレーズとして書き直す。

**NG例:**

```markdown
## 課題：認証の複雑さ

## 解決策：トークンベースの認証

## アーキテクチャ：マイクロサービス構成

## キャッシュ戦略 -- 読み取りは Redis、書き込みは DB
```

**OK例:**

```markdown
## 認証はなぜ複雑になるのか

## トークンベースの認証を導入する

## マイクロサービス構成にする

## 読み取りは Redis、書き込みは DB にするキャッシュ戦略
```

### 5-3. 中黒（・）による並列

日本語の単語の並列に中黒（・）を使わない。読点でつなぐか、箇条書きにする。ただし単一の固有名詞の内部（「マルクス・アウレリウス」など）では使ってよい。

**NG例:**

```markdown
設計・実装・テストを担当する
```

**OK例:**

```markdown
設計、実装、テストを担当する
```

### 6. コロン後のブロック要素（英語的パターン）

述語（動詞・形容詞）で終わる文の後にコロン＋ブロック要素は避ける。

**NG例:**

````markdown
実行します:

```bash
command
```

説明します:

- 項目1
- 項目2
````

**OK例:**

````markdown
実行方法は以下の通りです。

```bash
command
```

説明の内容は以下の通りです。

- 項目1
- 項目2
````

名詞で終わる場合は許容される:

```markdown
使用方法:
例:
```

## テクニカルライティングのベストプラクティス

### 簡潔性

| NG                   | OK                    |
| -------------------- | --------------------- |
| まず最初に           | まず                  |
| 〜することができます | 〜します / 〜できます |
| 〜を行う             | 〜する                |
| 〜という             | 〜                    |

### 明確性

- 受動態より能動態を優先する
- 「〜される」より「〜する」

### 具体性

| NG     | OK          |
| ------ | ----------- |
| 高速な | 50ms未満の  |
| 多くの | 10以上の    |
| 簡単に | 3ステップで |

### 構造化

- 長い文は分割する（目安: 1文60文字以内）
- 1つの段落に1つのトピック

## 段落と論証の構成

パラグラフライティングを基本とする。段落は論証の一歩であり、読者が段落単位で論理を追えるようにする。

- 1つの段落には1つのトピックだけを置く。調査・報告・検証・評価など複数の場面が混ざった長い段落は、一歩ずつの段落に分割する。
- 段落の最初の文を読めば、その段落が何の話かわかるようにする。
- 段落の先頭で、前の段落との論理関係を接続表現で示す（「であれば」「実際」「しかし」など）。
- 論証は一方向に進める。結論を出したあとで反論を処理し、また結論を言い直す構成にしない。疑念の処理を終えてから、結論を一度だけ置く。
- 読者が抱きそうな誤解は、明示的に否定してから本当の理由を述べる（「その理由は〜だからではない。〜だからだ」）。
- 「AではなくB」と否定するときは、否定の根拠を一文添える。
- 「後の章で扱う」のような前方参照は、論証が一段落した位置（段落末・節末）に置き、途中で流れを切らない。

## 論証を厳密にする

書き上げたら、読み手の反論を先回りして点検する。

推量・可能性・反実仮想として書いた文を、機械的に断定へ変えない。「かもしれない」「だろう」は、根拠なく主張を弱めている場合だけ削る。事実未確認、推定、読者が抱きそうな疑念を表す場合は、その不確実性を残す。

| NG                             | OK                         |
| ------------------------------ | -------------------------- |
| 提示し続けているかもしれない（根拠を確認できないのに断定に変える） | 提示し続けている可能性がある |

- 区別すべき対象（別々の決定、別々の原因、種類の違う問題）を一括りの言葉でくくらない。相互依存する3つの未決事項を「同じ決定を別々に下していた」ではなく、「どれも別々の決定であり、しかも互いに依存している」と腑分けする。
- 複数の要因がある事象を、単一の原因に還元しない。例が複数種類の問題を含むなら、それぞれを切り分ける。
- 因果を主張するときは、その機構（なぜそうなるのか）を一文で示す。「手順で分けると変更が全体に波及する」ではなく、「各工程がデータを受け渡す表現を共有してしまい、その表現を変えると全体に波及する」と書く。
- 検出・保証・解決を「必ず」できるかのように書かない。条件付きで正確に述べる（「〜しやすい」「〜できることが多い」「〜が成り立つときに限り」）。
- 章・節をまたいで、同じ概念の扱い・分類・用語を一致させる。

## 読み手の負荷を管理する

読者の記憶と注意は有限の資源として扱う。

- 後で参照しない固有名（ファイル名、関数名、識別子）を出さない。「仕様書」「金額計算のユーティリティ」のような一般的な言い方で済ませる。
- 抽象的な言い回しの指す内容が文脈から一意に決まらないときは、丸括弧による同格挿入でその場で特定し、読者に前を読み返させない。
- 新しい例や場面を追加するときは、前の例と何が違うのか、なぜもう一つ必要なのかを前置きする。

## 視点と語り

- 例示では、結果の羅列や受動態（「特定され、判明した」）ではなく、行為者を主語にした動作の連なり（「リポジトリを調査して特定した」）で書く。
- 「入社2年目のエンジニアが」のような架空の人物設定を無意味に冠しない。
- 読者を「あなた」と呼ばず、役割名（「開発者」「読者」）で書く。二人称の呼びかけは、導入や結びなど限られた要所にとどめる。
- 対象を指す語は具体的に選ぶ。「AI」「ツール」のような広い語でぼかさない。一度導入した術語は、以後も同じ語で通し、曖昧語に後退しない。
- 術語・訳語は、その分野で慣用されている語を選ぶ（プッシュ通知は「配送」ではなく「配信」など）。

## 演出を抑える

修辞は、それが効果を生む要所でのみ使う。全面禁止ではなく、節度の規範である。

- 溜め（「ここには〜が潜んでいる」）や修辞疑問で導出を演出するのは、緊張が議論に効く要所に限る。説明で足りる箇所はそのまま述べる。
- 短い決め台詞を独立した段落にして緊張を作る演出を多用しない。
- 「〜してはならない」という命令調より、「〜するわけにはいかない」のような作業者の判断として書く形を選ぶ。
- 転回点を過剰に劇的にしない。事実を述べる一文で足りる場合が多い。
- 「AではなくBだった」という対句の決め台詞を多用しない。

## LLM っぽい表現を避ける

論点を増やさず「ちゃんと書いている感」だけを付ける言い回しを使わない。書き上げたら、この観点で点検する。

- 予告と総括: 「重要なのは〜である」「本章では〜を扱う／探求する」「ここでは〜について見ていく」「まとめると」「要するに」（直前の言い換えだけのとき）、「〜に他ならない」
- 正面から系: 「正面から扱う」「正面から回収する」（中身の代わりに姿勢だけを宣言する）
- 空虚な形容: 「不可欠」「核心的」「鍵となる」「根本的な」「多角的」「包括的」「総合的」（中身を説明せず強調だけする）
- 空虚な動詞: 「掘り下げる」「深掘りする」「言語化する」「触れる」「言及する」
- 接続の型: 「〜において」「〜という側面から」「〜の観点から」（新情報がないとき）、「さらに」「また」「加えて」の連打
- 弱い緩和と称賛: 「〜と言えるだろう」（根拠なく弱める場合）、「非常に」「極めて」「大いに」（中身のない強調）

**NG例:**

```markdown
本章では、〇〇の理論を正面から扱う。
多角的に分析すると、重要なのは品質である。
```

**OK例:**

```markdown
本章では、〇〇の理論を扱う。
評価の核心は、正しさを誰が知っているかにある。
```

## 冗長を排除する

- 同じ主張を言い換えて繰り返さない。1つの主張は一度だけ書く。
- 隣接する節が同じことを別の角度で述べているなら、片方に吸収して1つにまとめる。
- 場面を描写した直後に、その内容を要約し直さない。
- 読者が自力で補える中間段階の説明は書かない。
- 接続や評価のためだけの文（「それ自体はよいことである」など）を置かない。
- 想像上の読者との問答（問いを立てて一語で答える形）を修辞として使わない。主張はそのまま述べる。

## 見出しは内容を特定する

- 作業の手順だけを述べる見出し（「例に戻す」「〜を読み直す」）や、情報量のない見出しにしない。その節が答える問い、または扱う対象を見出しにする。
- 見出しを、節の結論を言い切る「セリフ」にしない。見出しの時点で読者がオチを知る状態を避ける。
- 疑問形でも対象を指す名詞句でもよい。本文のトーンに合うほうを選ぶ。

## 読者への誠実さ

- 例が作為的に見えうる場合、それを隠さない。読者の疑念を先回りして認め、現実にあり得る根拠を短く添える。
- その根拠は、著者の断定ではなく、読者自身の経験に訴える一般的事実や通説に求める（「この症状は珍しくないだろう」など）。
- 確認していないことを、確認したかのように滑らかに書かない。
